# Evaluation Workflow

This file explains how the main evaluation agent should run one complete evaluation.

When a user asks you to run evaluation in this workspace, that request is considered permission to use Claude Code subagents (the Task / Agent tool). The main agent may start clean-context skill-generation subagents and solver subagents to generate skills and complete test attempts.

Solver and skill-generation subagents run with the same model and reasoning/effort setting as the main agent — this is inherited automatically and does not need to be set.

If the required number of subagents exceeds the subagent concurrency limit, run them in batches. Batched execution must still preserve clean context for every solver attempt, unique `eval_attempt_id` values, and complete run records. Do not reduce the attempt count, let one solver solve multiple test tasks, or let the main agent solve test tasks directly.

Before starting any skill-generation or solver subagent, the main agent must prepare a dedicated directory that holds only that subagent's allowed files, and instruct the subagent to read and write only inside that directory. Do not give a subagent access to the `eval_workspace` root, the `task_group/` root, or any directory that contains files outside that subagent's allowed information boundary.

## 1. Prepare The Task Group

This workspace evaluates one task group at a time. The task group under evaluation should be located at:

```text
task_group/<task_group_id>/
```

The task group must have passed quality review.

## 2. Check The Workspace

The main agent first confirms that the workspace contains exactly one task group and that it includes:

- 5 train tasks.
- 5 test tasks.
- A task-group-level shared environment.
- Official input, standard answer, and evaluator for each task.

## 3. Start The Shared Environment

The main agent prepares the task-group environment and summarizes the environment entrypoints visible to solvers.

If a Web/API service is needed, roll a random candidate port in `8000-8100`; if it is occupied, roll another one. Do not scan upward from `8000`. Record:

- Start command.
- Port.
- Solver-visible port, Web/API URL, or database connection instructions.
- Environment errors or restart records.

Skill-generation and solver subagents must not enter, list, or read `env/`. They may use the shared environment only through the port, Web/API URL, or database connection explicitly exposed by the main agent.

## 4. Generate Skills

The main agent generates 3 independent skills for each skill condition. Each skill must be generated by a clean-context skill-generation subagent using the `skill-creator` skill.

Each skill-generation subagent should be given a dedicated working directory, restricted to it, such as:

```text
scratch/skill_generation/demo_attempt_01/
scratch/skill_generation/reflect_attempt_01/
```

The main agent stages only the allowed train inputs, allowed train answers, and exposed environment entrypoints into that directory. For `reflect`, do not stage train answers until the blind train attempts have been saved. The skill-generation subagent writes its draft skill inside its own directory; the main agent then copies the accepted skill into `skills/`.

Recommended layout:

```text
skills/demo/demo_attempt_01/SKILL.md
skills/demo/demo_attempt_02/SKILL.md
skills/demo/demo_attempt_03/SKILL.md
skills/reflect/reflect_attempt_01/SKILL.md
skills/reflect/reflect_attempt_02/SKILL.md
skills/reflect/reflect_attempt_03/SKILL.md
```

Generation rules are defined in `skill_modes.md`.

Each generated skill should be a skill directory whose markdown entrypoint is `SKILL.md`.

Skill-generation token usage is not included in solver efficiency metrics.

## 5. Run The Base Experiment

Run each test task independently 3 times. The solver only receives the official input for that test task and the allowed environment entrypoints.

Recommended record layout:

```text
runs/base/test_001/attempt_01/answer.json
runs/base/test_001/attempt_01/score.yaml
runs/base/test_001/attempt_01/run_metadata.yaml
```

Launch each solver subagent restricted to the corresponding attempt directory:

```text
runs/base/test_001/attempt_01/
```

Before launch, stage only the allowed files into that attempt directory:

- `input/` copied from the current test task's official `input/`.
- `environment_access.md` or equivalent concise instructions for the exposed Web/API/database entrypoints.

Do not stage `env/`, task outputs, task notes, evaluator files, train tasks, other test tasks, generated skills, or prior run outputs for base runs. The solver writes `answer.json` in its own attempt directory.

## 6. Run The Demo Experiment

Run each test task independently 3 times. The solver receives the official input for that test task, allowed environment entrypoints, and the corresponding independently generated demonstration skill:

```text
attempt_01 uses skills/demo/demo_attempt_01/SKILL.md
attempt_02 uses skills/demo/demo_attempt_02/SKILL.md
attempt_03 uses skills/demo/demo_attempt_03/SKILL.md
```

Every run still requires clean context. Do not let one solver solve multiple test tasks.

Launch each solver restricted to its own attempt directory, for example:

```text
runs/demo/test_001/attempt_01/
```

Stage only the current test task `input/`, environment entrypoints, and a copy of the matching generated skill for that attempt number. Do not expose the full `skills/` directory or skills from other attempt numbers.

## 7. Run The Reflect Experiment

Run each test task independently 3 times. The solver receives the official input for that test task, allowed environment entrypoints, and the corresponding independently generated reflection skill:

```text
attempt_01 uses skills/reflect/reflect_attempt_01/SKILL.md
attempt_02 uses skills/reflect/reflect_attempt_02/SKILL.md
attempt_03 uses skills/reflect/reflect_attempt_03/SKILL.md
```

Every run still requires clean context. Do not let one solver solve multiple test tasks.

Launch each solver restricted to its own attempt directory, for example:

```text
runs/reflect/test_001/attempt_01/
```

Stage only the current test task `input/`, environment entrypoints, and a copy of the matching generated skill for that attempt number. Do not expose the full `skills/` directory or skills from other attempt numbers.

## 8. Score And Aggregate

After each solver writes `answer.json`, the main agent calls the corresponding task evaluator and writes `score.yaml`.

Every solver attempt must have a unique `eval_attempt_id`, and that ID must appear in the solver prompt, attempt directory, and `run_metadata.yaml`. Recommended format:

```text
<task_group_id>__<condition>__<task_id>__attempt_<nn>__<timestamp>
```

The main agent backfills token usage from the session transcript. Do not attribute usage using only "the newest entry"; confirm all of the following:

- The turns belong to a subagent invocation, not the main agent.
- The subagent was launched by the current main evaluation agent (its `parent_tool_use_id` matches the Agent tool call issued for this attempt).
- The attempt's `answer.json` was written under the expected attempt directory, such as `runs/<condition>/test_<nn>/attempt_<mm>/`.
- The subagent prompt contains the corresponding `eval_attempt_id`.

Each solver subagent has its own transcript:

```text
~/.claude/projects/<project>/<session-id>/subagents/agent-<agent_id>.jsonl
```

One API response is logged across several content-block records: `input`/`cache_creation`/`cache_read` are identical, but `output_tokens` is cumulative (streamed). **Deduplicate by `message.id`** — keep the input/cache buckets from any record and take the **max `output_tokens`** per `message.id`; summing every record over-counts input/cache ~2-3x and keeping the first record under-counts output. Sum the four buckets (`input_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`, `output_tokens`) across the deduped responses, and compute `cost_usd` with the per-response formula in `metric_and_scoring.md`. Do not use the parent's `toolUseResult.totalTokens` as the billed total — it is a context-size measure that excludes cache reads.

After all runs complete, aggregate `acc@3`, average per-bucket tokens, and average `cost_usd` for the three conditions. These efficiency metrics only count answer-writing by test solver subagents: first average the 3 attempts for the same test task, then average the 5 test tasks. Do not include skill generation, environment startup, evaluator execution, or main-agent summarization.

Temporary checking code, aggregation code, or environment startup notes may be placed under `scratch/`. These materials are not formal evaluation data.

## 9. Interpret Results

In the report, explain:

- Overall `acc@3` for all three conditions.
- Improvement from each skill condition over base.
- Whether reflection skill outperforms demonstration skill.
- Which test tasks improved clearly and which did not.
- Any environment instability, output-schema friction, evaluator issue, or suspicious leakage risk.
