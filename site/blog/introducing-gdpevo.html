GDPevo: Measuring Skill Transfer in Business Task Groups

A public benchmark for studying whether agents can turn related training work into reusable skills that improve held-out business tasks grounded in productivity-bearing business environments.

Abstract

Existing agent benchmarks have made substantial progress on realistic environments, tool use, and professional deliverables, but many still report performance at independent-task granularity. This leaves a practical capability under-specified: whether an agent can reuse procedures learned from related work and complete future business tasks more reliably and at lower downstream solving cost. GDPevo evaluates this gap through task groups. Each task group contains one shared business environment, five train tasks, five held-out test tasks, reference answers, exact evaluators, and notes. We compare three execution conditions: no_skill, demonstration_skill, and reflection_skill. The first public Codex GPT-5.5 xhigh run covers 12 task groups and reports avg@3 scores, solver token usage, and cost. Across the released run, skill conditions improve average test performance over the no_skill baseline while reducing average token and cost usage.

1. Introduction

Current agent benchmarks leave three practical questions difficult to answer. First, can an agent reuse procedures across related tasks? Second, does that reuse improve held-out work in the same environment, where records, entities, constraints, and output requirements change? Third, can the effect be measured with exact task scores and downstream cost signals, while preserving inspectable artifacts such as task notes and generated skills?

GDPevo turns these questions into a benchmark unit: the task group. A task group contains one shared business environment and a train/test split of related tasks. The solver may need to operate a web workspace, query an API or database, inspect business records, reconcile inconsistent files, understand domain rules, and return a JSON object or spreadsheet-like decision artifact.

Because train and test tasks share an environment, the benchmark can measure whether prior work becomes a portable operating method. It separates direct test-solving ability from skill-mediated transfer while keeping the environment, scoring target, and generated skills inspectable.

GDPevo treats skill transfer as a property of recurring business work. The task groups are not puzzles with business vocabulary; they are modeled after work products that would normally consume human business effort. A useful skill should not only raise correctness; it should also make future work more directed, reducing wasted search and lowering the marginal cost of similar tasks.

This treats agent learning as a measurable change in future behavior. A rising score over independent tasks can track general capability, but it does not isolate experience accumulation. In GDPevo, train tasks provide an explicit source of prior experience, and held-out test tasks measure whether that experience changes subsequent solving.

2. Data Construction Pipeline

The benchmark begins with scenario examples. A scenario example is not just a domain label; it is a concrete task pattern grounded in real business or benchmark work. Good examples reveal a reusable procedure, such as reconciling conflicting sources, applying eligibility rules, calculating a management metric, filtering stale records, or producing a structured operational decision.

01 scenario examples
02 shared environment
03 train/test tasks
04 quality review

From a scenario, a producer builds one shared environment and derives five train tasks plus five test tasks. The train tasks must expose useful procedure. The test tasks must require related procedure under changed records, wording, or output requirements. This creates the central design constraint: the tasks should be close enough for transfer but not so close that copying is rewarded.

Each task also includes notes describing the task definition, expected solution approach, likely failure modes, and evaluation criteria. These notes are not solver input during evaluation; they are released to make the data auditable. They help reviewers and readers understand what the task is designed to test and why the scoring points matter.

3. Task Group Format

A task group is the smallest release unit in GDPevo. This unit is larger than a prompt because skill transfer cannot be measured from a single task. A task group contains the environment, train tasks, test tasks, answers, evaluators, and notes needed to inspect the full transfer problem.

env/ Shared business environment and allowed access surface.
train_tasks/ Five related tasks used to derive reusable skills.
test_tasks/ Five held-out tasks used for final scoring.
input/ Prompt, payloads, and answer template visible to the solver.
output/ Reference answer used by evaluators.
eval/ Task-specific scoring logic.
notes/ Human-readable explanation of definition, method, errors, and criteria.

The shared environment is essential. It makes the benchmark about repeatable operational knowledge rather than isolated prompt reading. The solver may need to inspect a web application, call an API, query a database, or combine multiple sources, depending on the task group.

4. Skill Conditions

GDPevo evaluates the same test tasks under three conditions. The conditions differ only in whether a generated skill is provided and how that skill is produced.

A

no_skill

The solver receives only the test task input and allowed environment access. This condition measures cold-start task performance.

B

demonstration_skill

A skill is generated from train inputs and train reference outputs, then provided to the solver during test attempts.

C

reflection_skill

A skill is generated after blind train attempts, comparison against outputs, and reflection on the mistakes.

The distinction between demonstration and reflection is important. Demonstration skills test whether input/output pairs reveal a reusable procedure directly. Reflection skills test whether attempting the train tasks and correcting errors produces a better operational skill.

The public results pair avg@3 with token and cost metrics, so skill-conditioned gains can be read together with the cost of downstream solving. Because these tasks stand in for productive business work, cost reduction is part of the deployment signal rather than a cosmetic metric.

5. Evaluation Protocol

The primary metric for a task group is avg@3. For each skill condition, the evaluation runs five held-out test tasks, with three independent solver attempts per test task. The task-group avg@3 is the average score over those test attempts.

task_group_avg@3 = mean(score over 5 test tasks x 3 attempts)

Public result tables display avg@3 as a percentage.

Each solver attempt is context-clean and receives only the allowed task input, environment access information, and the skill required by the condition. The evaluation focuses on final test-solving performance. Efficiency metrics count solver answer-writing attempts: cached input tokens, input tokens, output tokens, and board-level cost. Token counts are displayed in thousands (k) in public tables.

Evaluators score structured business outcomes rather than free-form plausibility. Scoring points are tied to concrete results such as exact record sets, totals, rankings, dates, classifications, flags, or required JSON fields. This makes failures easier to interpret: a wrong score can be connected to missing retrieval, incorrect source precedence, arithmetic mistakes, or schema errors.

6. Public Release

The first public release contains 12 task groups, 60 train tasks, 60 test tasks, structured evaluation reports, and generated skill packages. The released task groups cover business environments such as CRM, ERP, finance, banking, HR, procurement, inventory, and operational analytics.

The repository is organized so that task data and experiment artifacts can be inspected together. Task groups live under data/task_groups/. The experiment board summarizes released scores and efficiency metrics. Each report YAML records condition-level and per-task results. Skill packages referenced by the reports are released under the corresponding experiment directory.

Generated skills are included because they are part of the measured behavior. They show what the skill-creation process extracted from the train tasks. Inspecting them helps explain why a condition improved, failed, or used more resources.

7. Results

The initial public run evaluates Codex with GPT-5.5 xhigh. Across 12 task groups, both skill conditions improve over the cold-start baseline.

no_skill 48.35%
demonstration_skill 65.99%
reflection_skill 67.13%

Demonstration skills improve the average score by +17.64 percentage points over no_skill. Reflection skills improve the average score by +18.78 percentage points. The largest observed task-group gain appears in task_group_009, where demonstration_skill improves from 42.76% to 92.47%.

Skill-conditioned solving also reduces average cost in the released run: demonstration_skill lowers cost by 24.8% and reflection_skill lowers cost by 26.7% relative to no_skill. These token and cost metrics extend the score result: in this run, better business-result accuracy comes with less average answer-writing work, which is the behavior we want from reusable skills on recurring business tasks.

The result should be read as a diagnostic table rather than a final model ranking. Some task groups favor demonstration skills because the train input/output pairs expose the recurring rule directly. Other task groups favor reflection skills because the useful procedure becomes clearer after the agent attempts train tasks and analyzes mistakes.

8. Discussion

The first release supports the basic premise of the benchmark: generated skills can improve performance on related business tasks while often reducing redundant downstream exploration. It also shows that skill transfer is not uniform. A task group may have small gains because the no_skill baseline is already strong, because test tasks require new exploration, or because the generated skill misses the important procedure. These cases are useful because they reveal different failure modes.

GDPevo is designed to make these differences inspectable. The task notes explain what each task is testing. The evaluators define the scoring target. The reports expose per-condition performance and efficiency. The generated skills show what was learned from train tasks. Together, these artifacts make it possible to study skill learning as a workflow rather than as a single aggregate score.
