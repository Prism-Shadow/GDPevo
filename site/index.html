Benchmark dossier

Measuring whether agents carry skills into real work.

GDPevo evaluates whether an agent can derive a reusable skill from training tasks and transfer it to related test tasks inside the same business environment. The tasks are grounded in productivity-bearing office work, so better transfer means more accurate business outcomes and lower downstream solving cost.

research question

Direct solving is not the whole story.

Many agent benchmarks ask whether a model can solve a single task. GDPevo asks what changes after the agent has seen related work. A task group creates that comparison by keeping the environment fixed while varying the business requests. The measurement target is whether train-derived skills help the solver complete later productivity-bearing business work more accurately, with less redundant search and lower answer-writing cost.

released results

The first run shows measurable transfer.

The public Codex GPT-5.5 xhigh run reports three skill conditions across 12 task groups. The board score is averaged over three independent attempts per test task and then over the five test tasks. Skill conditions improve accuracy and reduce average answer-writing token usage and cost, suggesting that train-derived skills can make later business work more directed and cheaper to execute.

construction system

A task group is built as a transfer instrument.

01 scenario examples

Start from related business or benchmark examples that expose a reusable work pattern.

02 shared environment

Build one operational surface with records, rules, APIs, files, or web interfaces.

03 train/test tasks

Create five train tasks and five test tasks that differ in request details but share procedure.

04 review and release

Check evaluators, notes, boundaries, and transfer design before reporting avg@3 results.

task group anatomy

One environment, two task sets, one score report.

The release unit is intentionally larger than a prompt. It contains the environment, prompts, answer schemas, reference outputs, evaluators, and notes required to inspect both the data and the scoring target.

evaluation protocol

Three conditions separate direct ability from transferable procedure.

condition A

no_skill

The solver receives the test task input and allowed environment access.

condition B

demonstration_skill

A skill is generated from train inputs and outputs, then reused for test attempts.

condition C

reflection_skill

A skill is generated after blind train attempts, output comparison, and reflection.

Benchmark design, pipeline, evaluation protocol, and first results.
