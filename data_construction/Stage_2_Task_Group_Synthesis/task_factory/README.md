# Task Factory Workspace

`task_factory/` is the English working specification for stage 2 benchmark construction. Starting from one stage 1 scenario and several examples under that scenario, the factory builds a train-predict task group with 5 train tasks and 5 test tasks. Both train and test tasks should be formal tasks drawn from the same real-task distribution. Train tasks are not tutorials, teaching problems, or simplified examples; their solver-visible inputs and standard answers serve as solved examples for fewshot skill generation.

This directory defines the workspace structure, construction requirements, and fixed calibration prompts. Formal calibration always uses the Codex harness with `gpt-5.5` at `xhigh` reasoning effort, regardless of the model running the construction workspace. Runtime placeholders such as owner, port, image, and run id must be filled in, but the calibration model, agent-facing material layout, and network topology are not left to the construction agent to choose.

## Directories

| Path | Purpose |
| --- | --- |
| `guides/` | Construction guides split by topic |
| `seed_scenario/` | One copied stage 1 scenario and its selected examples |
| `task_group/` | The task group under construction; keep the completed reviewed task group here |
| `scratch/` | Design drafts, data-generation records, calibration records, review notes, and intermediate artifacts |

## Core Principles

- One task group comes from one scenario and several examples under that scenario.
- A completed task group should contain 5 train tasks and 5 test tasks.
- Train and test tasks should come from the same real-task distribution. The train tasks are earlier calibration samples, not teaching problems.
- The task group must have both diversity and transferable SOPs, experience, and key facts.
- Diversity should stay inside a transfer band. Prefer 2-3 recurring operation families with varied entities, data scale, noise, source conflicts, and output schemas, rather than five unrelated SOP families that each appear only once.
- Each test task should include some high-weight scoring points that benefit from skills inferred from the train inputs and standard answers. Those transfer-dependent points should have clear train anchors; other high-weight points may come from task-specific exploration, data scale, or long-horizon work.
- Train tasks must not explicitly teach the SOP, expose easier versions of the test tasks, or reduce the benchmark to template copying. Transferable experience should be inferred from the solved train examples without copying task-specific final values into the skill.
- Every formal task should preserve long-horizon task complexity and difficulty aligned with the source examples. Do not make generated tasks substantially easier, narrower, or more toy-like than the examples.
- Large-scale data should be generated by programs and randomness, not handwritten.
- `env/` should be built as a shared public-data and office-work environment for the whole task group. It may include Web, API, or SQLite-backed business infrastructure, but it must not be partitioned into per-task data packages or per-task answer-like endpoints. Database-backed environments must use SQLite rather than PostgreSQL or another server database.
- Build the environment from `env/Dockerfile` and run it with every solver or skill-generation container on an owner- and run-scoped Docker bridge network. The environment binds `0.0.0.0` at `TASK_ENV_PORT = 9000 + task-group number`, uses the network alias `task-env`, publishes no host port, and is reached at `http://task-env:<TASK_ENV_PORT>/`. Declare `env.state_mode` as `read_only` or `mutable`: read-only instances may be shared within one capability stage, while mutable instances are fresh per attempt. Never mount `env/` into an agent container.
- Solver and test agents may use environment entry points such as URLs, API endpoints, or an authenticated SQLite query-service URL and credentials, but they must not inspect the `env/` directory, SQLite database files, source files, generated data files, seeds, manifests, or setup scripts directly.
- `env/endpoints.txt` must list every reachable endpoint as `METHOD /path` without descriptions. Calibration and evaluation inputs copy only the endpoint names allowed for the current run into `environment_access.md`.
- `env/` implementation should be handled by a clean-context env-builder coding subagent based on `scratch/env_blueprint.md`; the main agent owns the blueprint and final integration.
- Task files, including `input/`, `notes/`, `output/`, and `eval/`, should be produced by task-builder subagents for their assigned tasks. The main agent should not directly generate all task files and answers in one monolithic builder script.
- A script such as `scratch/build_task_group_*.py` must not create the shared environment, all 10 task folders, hidden standard answers, notes, evaluators, task-group index, scratch design docs, and calibration skill from one fixed specification. That is a workflow violation even if the generated files look structurally valid.
- Scripts are allowed only when their scope is narrow and explicit, such as shared `env/` data generation, one task-builder's local transformation or evaluator helper, or validation. They must not replace the required env-builder and task-builder subagent workflow.
- Generate 3 independent fewshot skill packages using 3 isolated processes. Each generator receives the 5 train inputs, the matching train `output/answer.json` files, and the environment entrypoint, then writes one complete package under `scratch/train_skill/fewshot_attempt_<nn>/` with `SKILL.md` as its entry file.
- Difficulty calibration must not use orchestration subagents as solvers. Every fewshot skill-generation, base, and fewshot run is a separate Dockerized `codex exec` process using `gpt-5.5` with `xhigh` reasoning effort, a dedicated staged `/work`, temporary `CODEX_HOME`, fixed launch prompt, and preserved raw trace. Never inherit or substitute the construction agent's model for calibration.
- Overall base `avg` should be about `0.40-0.60`; fewshot should add about `0.10-0.30` while remaining roughly below `0.80` overall and avoiding broad score saturation.
- `notes/notes.md` is the interpretability file for each task. It records the problem definition, solution method, transfer source, model pitfalls, evaluation criteria, and data-generation notes.
- `notes/notes.md` should be bilingual in English and Chinese for human review. It is the only final task-group artifact that should contain Chinese; solver-visible inputs, answer templates, standard answers, evaluators, task metadata, and environment files should remain English-only.
- Every train and test task must include `input/payloads/answer_template.json`, which defines the required output JSON shape, field types, numeric precision, and allowed enum choices.
- Each task should normally contain 6-10 scoring points spanning at least 4 distinct business questions or outcomes. Points should avoid using different wording to reward the same underlying criterion, answer fact, or root decision more than once. Splitting one decision into correlated or semantically duplicate rows is not multidimensional evaluation.
- Each point has raw weight `1`, `2`, or `3`, converted into that point's assigned score by `weight / sum(weight)`. A point earns all of its assigned score when its complete goal passes, otherwise it earns zero. Do not award fractions within a point. Genuinely different outcomes that can be wrong separately should use separate meaningful points, but rephrasing the same outcome must not create extra credit.
- Scoring points should prefer numeric, enum, boolean, ranking, set, or normalized structured results. If a scored field would otherwise require string matching, convert it into a controlled-choice field in `answer_template.json` to avoid schema friction.
- Most scoring points must depend on transfer learning, substantial data exploration, or long-horizon work. Base attempts should not obtain most points by simply reading the prompt or filling a format.
- Solver-visible `prompt.txt` and `input/payloads/` must not plainly leak SOPs, key facts, tool workflows, or step lists such as `(1)(2)(3)(4)`.
