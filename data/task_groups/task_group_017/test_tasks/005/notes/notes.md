## 中文

数据来源：本任务属于 `task_group_017`，来自 `SCN_017_white_collar_investigation_production_review`，参考源示例 `E001`、`E002`、`E003`。任务只使用共享环境 `task_group/task_group_017/env/` 中的生成数据，包括事项、传票类别、生产日志、收集事件、保留规则、销毁事件、特权日志、质检事件、保管人和文档摘要。面向求解器的本地载荷只有 `answer_template.json`。

任务定义：求解器需要为 `M-ALD-507`、`M-BAY-144`、`M-LYN-322` 制作跨事项整改看板。可见输入只要求按照 JSON 模板输出排序后的整改优先级、问题到动作矩阵、跨事项共性失败模式，以及期限风险和通知风险布尔值。提示中不提供 SOP 清单或答案路径。

场景契合：该任务模拟律所或公司法务在多个调查事项之间做整改排程。它把生产缺口、保留与保全时间点、保管人收集缺陷、特权审查缺陷和质检缺陷合并成结构化决策板，符合白领调查中的生产审查和补救工作流。

材料映射：`matters.json` 提供事项、期限、保全日期和监管通知标记；`subpoena_categories.json` 提供类别 ID；`production_logs.json` 提供类别状态和数量；`collection_events.json` 提供缺失来源和收集状态；`destruction_events.json` 区分保全前政策销毁和保全后灭失；`privilege_logs.json` 提供特权缺陷；`qc_events.json` 提供误编码、共享文件恢复、特权误编码和附件失败；`documents.json` 提供关键文档锚点。

答案与评测依据：可见答案模板允许求解器自定义稳定 issue ID。隐藏标准答案现在使用中性的 `ISSUE-###` 代码；评测器不要求隐藏语义 ID 字符串，而是先按事项、类别、问题类型、时间分类、数量和主动作等业务字段匹配 `matter_action_matrix` 行，再使用求解器自己的 issue ID 检查 ranked priorities、cross-cutting failures 和 deadline-risk 行之间的一致性。关键事实包括 Alderline 个人消息需监管通知和补充收集、Alderline 特权日志缺口为 1,365、Bay Teams 为保全后灭失且 11 个频道不可恢复、Bay 实验室数据是保全前政策缺口、Bay Tidewater 审计报告需供应商取回、Lynxion 共享文件 52 个受影响且 11 个未恢复、Lynxion 特权误编码需 clawback 检查、失败附件需重新质检。评测器有 9 个结构化评分点，原始权重为 `[1,3,3,3,3,3,2,2,1]`，输出 JSON 分数并自检标准答案应为 1.0。

迁移设计与训练锚点：这是测试任务。`train_001` 锚定首次生产缺口、个人渠道、监管通知和特权日志差额；`train_002` 锚定保留时间、保全前/后销毁、归档或供应商可恢复来源和 hold refresh；`train_003` 锚定设备擦除、共享来源删除、个人邮件、特权豁免、过度标注、clawback 和附件失败；`train_004` 锚定投诉误编码、未记录特权数量、律师通信过度标注和 clawback；`train_005` 锚定来源收集就绪、hold 遗漏、归档验证和动作排序。SP002 到 SP008 依赖这些迁移经验，同时还要求求解器在三个新测试事项中排除噪声并整合证据。

常见陷阱：模型可能误用 `AL-N`、`BA-N`、`LY-N` 噪声类别；把 Bay 的保全前实验室数据销毁误判为保全后灭失；漏掉 VaultSeven 或 board portal 是可恢复外部来源；漏算 Alderline 未记录特权数量；把 Lynxion 特权误编码只判为 privilege review 而不是 clawback；或输出叙述性文字。
